# 一、项目目标

开发一个“企业微信内使用的 FAQ 智能整理助手”：
- 运行形态：企业微信 H5 应用
- 主要用户：资料提交人、业务审核人、知识管理员
- 核心流程：上传资料 -> 自动解析清洗 -> 大模型生成 FAQ 候选 -> 人工审核 -> 发布为正式 FAQ
- 使用场景：传统制造企业 / 空调配件企业内部知识整理

# 二、硬性要求

## 1. 前端要求
- 技术栈：Vue 3 + Vite + TypeScript + Vant + Pinia + Vue Router + Axios
- 必须是手机端优先设计，适配企业微信内嵌浏览器
- 页面风格简洁、卡片化、单列布局、适合 375px 宽度
- 不做复杂 PC 后台
- 所有主要操作按钮适合手指点击
- 页面必须包含基础 loading / empty / error 状态

## 2. 后端要求
- 技术栈：Java 17 + Spring Boot 3 + Spring Web + Spring Security + JWT + JPA（或 MyBatis Plus 二选一，但要统一）+ PostgreSQL + Redis + MinIO
- 代码风格要工程化，分层清晰
- 提供 RESTful API
- 提供统一响应体、全局异常处理、日志
- 可通过 application.yml 配置企业微信、数据库、Redis、MinIO、大模型 API 地址等

## 3. 企业微信接入要求
- 实现企业微信 OAuth 登录流程
- 适用于“企业微信内打开 H5 获取员工身份”
- 需要包含以下能力：
  1. 生成 OAuth 跳转地址
  2. 前端处理 code / state
  3. 后端根据 code 获取企业微信用户身份
  4. 建立本地用户体系并签发 JWT
  5. 角色权限控制
- 代码中请预留“可信域名 / 回调地址 / corpId / secret / agentId”等配置项
- 要考虑 state 防 CSRF、token 缓存、code 一次性消费、用户首次登录自动注册

## 4. FAQ 能力要求
- 支持上传 PDF / Word / Excel / txt / csv
- 支持手工粘贴文本
- 上传后可触发 FAQ 候选生成
- FAQ 候选包含：
  - question
  - answer
  - category
  - keywords
  - sourceSummary
  - confidence
- 支持人工审核操作：
  - 通过
  - 编辑后通过
  - 驳回
  - 合并到已有 FAQ
- 审核通过后生成正式 FAQ

## 5. 大模型接入要求
- 不绑定具体厂商，使用通用 HTTP API 方式接入本地大模型
- 封装一个 LlmClient
- 预留模型接口配置：
  - baseUrl
  - apiKey
  - modelName
  - timeout
- 实现“根据文本 chunk 生成 FAQ 候选”的调用
- 强约束模型输出 JSON
- 需要有容错处理：模型返回非 JSON、空响应、字段缺失时要处理

# 五、实现细节要求

## 1. 登录与权限
- 用户首次通过企业微信登录时，如果本地无账号则自动创建
- 角色先采用默认 reviewer 或 submitter，可通过数据库初始化
- 所有业务接口都要能拿到当前登录用户

## 2. 文件上传
- 上传文件后保存原文件到 MinIO
- 数据库存文件元数据
- 支持限制文件大小
- 校验后缀白名单

## 3. 文档解析
- PDF：优先文本解析
- Word：支持 docx
- Excel：抽取表格单元格为文本
- txt/csv：直接读取
- 对解析结果做清洗：
  - 去空行
  - 去重复空白
  - 去明显页眉页脚噪音
- 切块时保留：
  - fileId
  - chunkIndex
  - 原始内容
  - 清洗内容
  - metadata

## 4. FAQ 生成
- 一份文件可生成多个 chunk
- 每个 chunk 生成 0~3 条 FAQ 候选
- 模型输出需解析成结构化对象
- 候选 FAQ 进入待审核状态
- 记录来源文件和来源 chunk

## 5. 审核逻辑
- approve：直接通过，生成正式 FAQ
- edit-approve：允许编辑问题和答案后通过
- reject：记录驳回原因
- merge：关联到已有 FAQ，并保留来源关系

## 6. 移动端交互
- 首页展示：
  - 待审核数量
  - 今日处理数
  - 快捷入口
- 审核详情页底部固定操作栏
- 长文本支持折叠展开
- 所有列表支持下拉刷新或“加载更多”的简单实现
